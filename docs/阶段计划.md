# 摔倒告警事件隐私保护 Demo 开发计划

## 1. 项目目标与边界

本阶段目标是完成“摔倒告警事件的视频隐私保护与受控证据调阅 Demo”。

YOLO 与 VLM 继续作为整体的摔倒检测模块，本阶段不调整模型结构、不优化检测性能、不做 RTSP 接入、不实现多节点防删除。摔倒检测模块输出结果统一映射为：

- `high_risk`：高风险摔倒告警；
- `low_risk`：低风险摔倒告警；
- `no_alarm`：不产生护工告警。

所有 `high_risk` 和 `low_risk` 事件均进入同一处置流程：立即通知护工、重复提醒、护工点击“已处理”后停止提醒并变为绿色。两者仅在颜色、提醒间隔和紧急程度上不同：

- 高风险：红色，默认每 20 秒提醒一次；
- 低风险：黄色，默认每 60 秒提醒一次；
- 已处理：绿色，停止提醒。

本阶段隐私展示统一采用：

> 原始背景保留；画面中所有检测到的人物替换为完整、不透明的黑色或深色剪影；默认移除原始音频。

不做背景模糊，不做骨架、Avatar、马赛克等其他模式。

原始事件片段只允许用于检测模块内部、隐私渲染、证据加密和后续受控调阅。普通护工只能查看剪影预览，不得查看、下载、获取路径或解密原始视频。

------

## 2. 目标业务流程

```text
YOLO + VLM 摔倒检测模块
        ↓
high_risk / low_risk
        ↓
立即创建告警并通知护工
        ├─ 高风险：红色，20 秒重复提醒
        └─ 低风险：黄色，60 秒重复提醒
        ↓
并行异步执行隐私与证据处理
        ├─ 生成 privacy_preview.mp4
        ├─ 加密原始事件片段为 evidence.enc
        └─ 删除原始明文事件片段
        ↓
护工查看剪影预览
        ↓
护工点击“已处理”
        ↓
告警变绿，停止重复提醒
        ↓
仅在事故调查、纠纷处理或管理审查时
通过双人审批临时解密查看原始证据
```

告警链路优先级最高。隐私剪影生成失败、加密失败、审计失败均不得阻止告警创建、告警提醒或护工点击“已处理”。

## 3. 阶段一：现有事件输出与访问边界整改

### 目标

先阻止原始事件视频被普通前端、静态文件服务或事件接口直接访问。

### 修改内容

1. 建立新的目录边界：

```text
data/
├── private_staging/
│   └── <event_id>/
│       └── raw_event.mp4
├── privacy_previews/
│   └── <event_id>/
│       └── privacy_preview.mp4
├── secure_evidence/
│   └── <event_id>/
│       ├── evidence.enc
│       └── evidence_meta.json
└── runtime/
    └── evidence_playback/
        └── <temporary_token>/
```

1. 原始事件片段不得保存到前端可直接访问的 `data/events`、`static` 或公开媒体目录。
2. 检查并修改 `app.py`、静态资源路由、事件详情接口、回放接口，确保普通接口不返回：

```text
raw_clip_path
clip_path
staging_plaintext_path
evidence_path
原始视频文件名
原始视频 URL
```

1. 清理旧目录中用于演示的原始 MP4，或迁移到私有目录；不能继续让旧事件通过历史链接直接访问。

### 验收标准

- 在浏览器、普通 API、静态资源目录中无法访问原始 MP4；
- 普通事件详情接口只返回告警信息和剪影预览状态；
- 原始事件片段只存在于后端私有目录中。

## 4. 阶段二：数据模型与风险语义改造

### 目标

将“模型结果”“护工处置”“隐私处理”“证据状态”拆开，避免状态互相混淆。

### events 表新增字段

```text
risk_level:
high / low / null

alert_status:
pending / handled / null

handled_by:
处理护工账号

handled_at:
处理时间

last_notified_at:
上次提醒时间

next_remind_at:
下次提醒时间

reminder_count:
已提醒次数

privacy_preview_status:
pending / processing / ready / failed

privacy_preview_path:
仅保存剪影视频私有相对路径

evidence_status:
pending / encrypting / encrypted / failed / deleted
```

不建议长期保存 `staging_plaintext_path`。该路径只存在于运行时任务上下文；加密完成并删除明文后立即清空。

`evidence_path` 只可由后端私有证据服务使用，不能进入普通 API 返回内容。

### 旧状态迁移

```text
confirmed_fall
    → high_risk

need_human_review / uncertain
    → low_risk

rejected
    → no_alarm
```

前端和业务文案中删除：

```text
确认摔倒
人工审核模型结果
```

统一替换为：

```text
高风险摔倒告警
低风险摔倒告警
现场核验告警
```

### 验收标准

- 模型结果不再被表述为“事实确认”；
- 高风险、低风险、已处理状态在数据库和前端一致；
- 旧字段仅兼容读取，不作为新的唯一事实来源。

## 5. 阶段三：告警与提醒状态机

### 目标

实现红黄告警、重复提醒、护工已处理和绿色状态。

### 后端修改

在 `run_vlm_worker.py` 完成 VLM 判定后：

```text
high_risk：
创建红色待处理告警；
next_remind_at = 当前时间；
默认间隔 20 秒。

low_risk：
创建黄色待处理告警；
next_remind_at = 当前时间；
默认间隔 60 秒。

no_alarm：
不创建护工告警；
后续删除原始事件片段。
```

新增独立提醒调度进程，例如：

```text
run_alert_dispatcher.py
```

其职责：

1. 周期性查询 `alert_status = pending` 且已到 `next_remind_at` 的事件；
2. 更新提醒次数、上次提醒时间和下一次提醒时间；
3. 前端轮询或接收更新后播放提示音、闪烁红黄卡片；
4. 对已处理事件不再发送提醒。

### 前端修改

告警页面统一显示：

```text
高风险摔倒告警：红色
低风险摔倒告警：黄色
已处理：绿色
```

每条待处理告警包含：

```text
摄像头/区域
事件时间
风险等级
隐私视频状态
提醒次数
“已处理”按钮
```

点击“已处理”后：

```text
alert_status = handled
handled_by = 当前登录护工
handled_at = 当前时间
停止后续提醒
前端变为绿色
```

### 验收标准

- 高风险与低风险均会通知护工；
- 高风险默认 20 秒重复提醒，低风险默认 60 秒重复提醒；
- 护工处理后提醒停止；
- 已处理后不会再次出现红黄重复提醒；
- 提醒间隔可在配置文件调整。

## 6. 阶段四：剪影隐私预览生成

### 目标

生成普通护工可查看的隐私视频。

### 新增模块

```text
services/privacy_renderer.py
services/privacy_pipeline.py
```

### 处理逻辑

输入：原始事件片段。

输出：

```text
privacy_preview.mp4
```

处理要求：

1. 使用人体实例分割模型或兼容当前环境的人体分割模型；
2. 获取每帧中所有人物的人体掩码；
3. 对掩码做适度膨胀，减少头发、手脚、衣物边缘泄露；
4. 将所有人物区域填充为完整不透明黑色或深色剪影；
5. 背景保持原始画面，不做模糊；
6. 去除原始音频，输出静音剪影视频；
7. 不在剪影预览中保留原始人物像素；
8. 处理异常时将状态设为 `failed`，但不展示原始视频。

### 失败处理

```text
剪影生成成功：
privacy_preview_status = ready

剪影生成中：
privacy_preview_status = processing

视频解码失败、模型异常、生成文件不可读等：
privacy_preview_status = failed
```

剪影失败时，前端显示：

```text
隐私视频暂不可用，请立即到场核验。
```

告警仍继续存在、继续提醒，不能因剪影失败而停止。

### 验收标准

- 护工默认只能播放剪影视频；
- 视频中所有已识别人物均为完整不透明剪影；
- 背景保留，原始音频移除；
- 剪影失败时绝不回退展示原始视频；
- 剪影失败不影响高风险或低风险告警。

## 7. 阶段五：原始证据加密与明文清理

### 目标

原始事件视频不长期明文保存，仅保留不可直接播放的加密证据。

### 新增模块

```text
services/crypto_service.py
services/evidence_service.py
```

### 加密流程

```text
原始事件片段 raw_event.mp4
        ↓
AES-256-GCM 加密
        ↓
evidence.enc
        ↓
加密结果校验
        ↓
删除 raw_event.mp4
```

要求：

1. 使用 AES-256-GCM；
2. 每次加密生成新的随机 nonce；
3. 绑定 `event_id`、版本号等 AAD；
4. 密钥从环境变量或本地非 Git 配置读取；
5. 密钥不得写入代码、Git、普通数据库字段、事件 JSON 或前端接口；
6. 保存私有加密元数据：

```text
algorithm
key_id
nonce
aad_version
created_at
ciphertext_sha256
```

1. 加密后应通过一次解密校验或哈希校验确认文件可恢复；
2. 校验成功后删除明文原始事件片段。

### 加密失败处理

```text
加密失败：
记录审计日志；
evidence_status = failed；
进入有限次数重试；
不影响告警；
不允许原始视频进入前端；
超过配置的明文临时存活时间后自动清理。
```

### no_alarm 处理

```text
VLM 输出 no_alarm：
不创建护工告警；
删除原始事件片段；
默认不长期保留 evidence.enc；
只保留无身份统计信息。
```

### 验收标准

- 高风险和低风险事件均存在 `evidence.enc`；
- 原始明文事件视频不再长期存在；
- 普通前端无法播放或下载 `evidence.enc`；
- 密钥不进入仓库与事件 JSON；
- 加密失败时会记录异常，但告警不受影响。

## 8. 阶段六：账号、双人审批与受控证据调阅

### 目标

普通护工只看剪影；原始视频仅可在双人审批后短时查看。

### 账号与角色

最低实现三个角色：

```text
nurse
security_admin
privacy_approver
```

要求：

- 护工可查看告警、播放剪影、点击已处理；
- 安全管理员可发起原始证据调阅申请；
- 隐私审批员可批准或拒绝；
- 申请人与审批人必须是不同账号；
- 角色必须由服务端 Session 或登录态校验，不能由前端传入用户名伪造。

### 新增表

```text
users
evidence_access_requests
audit_logs
```

`evidence_access_requests` 至少包含：

```text
request_id
event_id
applicant_id
purpose
status
approver_id
approved_at
expires_at
```

### 调阅流程

```text
安全管理员发起申请并填写理由
        ↓
隐私审批员批准
        ↓
服务端生成短时播放令牌
        ↓
临时解密 evidence.enc
        ↓
生成带动态水印的临时播放视频
        ↓
播放结束或令牌过期
        ↓
删除临时明文文件
```

动态水印至少包括：

```text
查看账号
查看时间
事件 ID
仅限授权调查使用
```

原始证据不得通过公开静态 URL 提供下载。

### 验收标准

- 护工无法访问原始证据；
- 申请人与审批人不能是同一账号；
- 未审批、过期或被拒绝时无法播放原始证据；
- 临时播放视频带水印；
- 令牌过期后不能继续访问；
- 临时明文文件被自动清理。

## 11. 最终 Demo 验收流程

现场演示至少覆盖以下情况：

1. VLM 输出高风险事件，护工立即看到红色告警；
2. VLM 输出低风险事件，护工立即看到黄色告警；
3. 红色每 20 秒、黄色每 60 秒重复提醒；
4. 护工点击“已处理”后，告警变绿色并停止提醒；
5. 剪影视频异步生成，护工只能看到“原始背景 + 全身不透明剪影”的静音视频；
6. 剪影生成失败时，告警仍继续提醒，页面显示“隐私视频暂不可用，请立即到场核验”；
7. 原始事件片段加密为 `evidence.enc`，明文片段被删除；
8. 普通护工无法访问原始视频；
9. 安全管理员申请、另一审批员批准后，才能短时带水印查看原始证据；
10. 播放到期后临时解密文件被删除，并可在审计日志中查看完整操作记录。
