# 摄像头实时摔倒监测演示前端设计文档

## 1. 目的

本文档定义 `fall_edge_gateway` 项目的第一版前端。

该前端用于导师对外展示和拉项目，定位为“养老照护摄像头实时摔倒监测演示平台”。首页使用项目介绍页，先讲清楚项目价值、应用场景、当前能力和后续扩展方向；摄像头监控工作台作为一个独立导航模块，和告警中心、事件记录等模块一样可点击查看。

当前底层原型仍然使用本地视频模拟摄像头输入，但前端表达应以“摄像头、告警、事件、回放、复核”为主语。本地视频只作为模拟摄像头流的实现方式说明，不作为页面叙事中心。

前端需要帮助外部受众快速理解：

- 项目解决的养老照护问题是什么。
- 系统面向摄像头风险监测，而不是单纯的视频文件检测工具。
- 当前原型如何用本地视频模拟摄像头输入。
- 摄像头如何产生疑似摔倒告警。
- 已确认摔倒事件如何被记录和查看。
- 待复核告警如何进入后续人工复核流程。
- 有哪些回放案例和评估指标可以支撑项目介绍。
- 哪些能力已经实现，哪些能力属于后续扩展方向。

## 2. 产品定位

推荐前端名称：

```text
养老照护摄像头实时摔倒监测演示平台
```

推荐副标题：

```text
基于 YOLO 候选检测与 Video VLM 复核的摄像头风险告警原型
```

第一版前端应当呈现为“项目介绍页 + 摄像头监控工作台 + 告警/事件/评估模块”。首页负责项目叙事，实时监测页负责体现摄像头监控系统形态。

## 3. 假设

- 现有 Python 检测流水线仍然是数据和行为的真实来源。
- 当前原型使用本地视频模拟摄像头输入，不持续提供实时画面。
- 后续接入 RTSP 或真实摄像头后，实时监测页的摄像头卡片可替换为持续实时画面。
- 第一版前端只读，不修改事件数据。
- 前端读取本地已有输出，不在网页中启动 YOLO 或 VLM 任务。
- 摄像头卡片数量先从事件数据中的 `camera_id` 自动生成。
- 事件状态优先使用 SQLite 数据库中的状态，JSON `category` 作为补充。
- 项目可能暂时没有事件数据；空状态必须清晰，页面不能崩溃。
- 隐私保护、哈希校验、防删除取证、授权查看是后续方向。前端可以描述这些方向，但不能把它们呈现为已经完成的交互功能。

## 4. 第一版不做

第一版明确不包含：

- 用户登录。
- 角色权限。
- 网页上传视频。
- 网页启动 YOLO 或 VLM。
- 真实 RTSP 接入。
- 持续实时视频流播放。
- 实时告警推送。
- 人工审核通过/拒绝按钮。
- 真实视频加密或解密。
- 真实哈希链校验。
- 删除恢复。
- 多节点存储操作。
- 后台管理工作流。

这些内容可以作为后续项目扩展方向展示，但不能作为第一版可操作功能出现。

## 5. 信息架构

导航包含七个部分：

```text
项目介绍
实时监测
告警中心
摔倒事件记录
待复核告警
案例回放与评估
技术方案
```

推荐演示顺序：

```text
项目介绍 -> 实时监测 -> 告警中心 -> 摔倒事件记录 -> 待复核告警 -> 案例回放与评估 -> 技术方案
```

首页是“项目介绍”。“实时监测”是同级导航模块，不作为首页强制展示。这样既适合导师对外讲项目，也保留摄像头监控工作台的真实系统感。

## 6. 页面设计

### 6.1 项目介绍

目标：让导师开场时快速讲清楚项目价值和系统形态。

展示内容：

- 项目名称和副标题。
- 一句话定位：

```text
面向养老照护场景，通过摄像头风险监测自动发现疑似摔倒告警，并保存事件回放与元数据，为后续隐私保护和可信取证提供基础。
```

- 核心能力卡片：
  - 摄像头风险监测。
  - 疑似摔倒告警生成。
  - VLM 复核降低误报。
  - 摔倒事件记录与回放。
  - 待复核告警管理。
  - 后续隐私保护与可信留存。

- 场景痛点：
  - 摔倒发现不及时。
  - 普通监控被动低效。
  - 照护视频隐私敏感。
  - 事件证据可信度不足。

- 当前实现状态：
  - 已实现：本地视频模拟摄像头输入、YOLO 候选检测、事件前后帧缓存、异步 VLM 复核、事件片段保存、元数据保存、指标报告生成。
  - 后续方向：RTSP/真实摄像头、持续实时画面、隐私保护、哈希校验、防删除证据留存、授权查看。

边界：

- 首页不堆事件表格。
- 首页不假装展示实时摄像头流。
- 首页可以提供“进入实时监测”的入口按钮或导航链接。

### 6.2 实时监测

目标：作为独立模块展示摄像头监控工作台，让外部受众看到系统接入真实摄像头后的使用形态。

该页面不能只是几个摄像头画面。摄像头卡片是主体，但页面还必须包含状态栏、最新告警、待处理摘要和模拟模式说明。

#### 6.2.1 顶部状态栏

展示：

- 摄像头数量。
- 当前风险摄像头数量。
- 已确认摔倒事件数量。
- 待复核告警数量。
- 疑似摔倒告警数量。
- 已过滤误报数量。

这些数字不作为“历史累计总数据”重点讲解，只用于当前页面状态摘要。

#### 6.2.2 摄像头监测区

摄像头卡片数量：

- 从事件数据中的不同 `camera_id` 自动生成。
- 第一版不依赖独立摄像头配置文件。
- 如果没有事件数据，显示“暂无摄像头事件数据”的空状态。

每张摄像头卡片展示：

- 摄像头 ID。
- 区域/房间：第一版没有配置时显示“模拟区域”。
- 在线状态：第一版显示“模拟在线”。
- 当前风险状态。
- 最近告警时间。
- 最近告警类型。
- 待复核数量。
- 已确认事件数量。
- 最近告警回放入口。

画面区域：

- 第一版使用占位画面，不假装存在实时流。
- 占位文案：

```text
当前为模拟摄像头
暂无实时画面
最近告警可在回放中查看
```

- 后续接入真实摄像头后，该区域替换为 RTSP 或其他实时视频流画面。

摄像头状态优先级：

```text
已确认摔倒 > 待复核 > 疑似摔倒 > 正常
```

状态映射：

```text
confirmed_fall
  -> 已确认摔倒

need_human_review
  -> 待复核

candidates
  -> 疑似摔倒

没有相关事件
  -> 正常
```

如果同一个 `camera_id` 下同时有多种状态，卡片主状态显示最高风险状态。

#### 6.2.3 最新告警侧栏

展示最近若干条告警：

- 告警时间。
- 摄像头 ID。
- 区域/房间。
- 告警状态。
- YOLO 分数。
- VLM 状态。
- 查看回放入口。

这里的“告警”包含：

- `confirmed_fall`
- `need_human_review`
- `candidates`

不默认展示 `rejected`。

#### 6.2.4 待处理摘要

展示：

- 待 VLM 复核数量。
- 需人工复核数量。
- VLM 队列状态，如果 `data/records.db` 可用。
- 当前是否存在高风险摄像头。

第一版只展示状态，不提供处理按钮。

#### 6.2.5 模拟模式说明

必须明确说明：

```text
当前原型使用本地视频模拟摄像头输入，因此本页暂不展示持续实时画面；接入 RTSP 或真实摄像头后，可替换为实时视频流。
```

### 6.3 告警中心

目标：展示系统发现的风险告警，让前端更像实时监测系统，而不是离线结果查看器。

数据范围：

- `confirmed_fall`
- `need_human_review`
- `candidates`

默认不展示：

- `rejected`

列表字段：

- 告警 ID / 事件 ID。
- 告警时间。
- 摄像头 ID。
- 区域/房间。
- 告警状态：已确认摔倒 / 待复核 / 疑似摔倒。
- YOLO 分数。
- VLM 结果。
- VLM 置信度。
- 查看回放。

筛选项：

```text
全部告警
已确认摔倒
待复核
疑似摔倒
```

点击告警后展示回放详情：

- 视频回放播放器。
- 摄像头 ID。
- 来源模拟视频。
- 告警时间。
- YOLO candidate 摘要。
- VLM 结果。
- VLM 判断理由。
- 可见证据。
- 隐私状态。
- 完整性状态。
- 留存状态。

### 6.4 摔倒事件记录

目标：展示已经确认的摔倒事件，形成接近真实业务系统的“事件档案”模块。

数据范围：

- 只展示 `confirmed_fall`。
- 优先使用 SQLite `events.status` 判断事件是否为 `confirmed_fall`。
- 如果数据库不存在，或事件没有数据库记录，再使用 JSON `category` 判断。

列表字段：

- 事件 ID。
- 摄像头 ID。
- 区域/房间。
- 事件时间。
- YOLO 分数。
- VLM 结果。
- VLM 置信度。
- 片段时长。
- 隐私状态。
- 完整性状态。
- 留存状态。

详情区：

- 视频回放播放器。
- 事件 ID。
- 摄像头 ID。
- 来源模拟视频。
- Clip 路径。
- YOLO candidate 摘要。
- VLM 判断结果。
- VLM 置信度。
- VLM 判断理由。
- 可见证据。
- 隐私状态。
- 完整性状态。
- 留存状态。

空状态：

```text
当前暂无已确认摔倒事件。
```

后续扩展说明：

- 该模块后续适合加入访问控制。
- 第一版只展示状态，不实现登录、授权或解密。

### 6.5 待复核告警

目标：展示还不能直接归档为摔倒事件、需要继续 VLM 复核或人工判断的告警。

数据范围：

- 展示 `candidates`。
- 展示 `need_human_review`。
- 不展示 `rejected`。

状态含义：

- `candidates`：YOLO 已发现疑似摔倒告警，可能尚未完成 VLM 复核。
- `need_human_review`：VLM 无法明确判断，需要人工审核。

列表字段：

- 告警 ID / 事件 ID。
- 当前状态。
- 摄像头 ID。
- 区域/房间。
- 告警时间。
- YOLO 分数。
- VLM 结果，如果已有。
- VLM 置信度，如果已有。
- 片段时长。

详情区：

- 视频回放播放器。
- 告警 ID。
- 摄像头 ID。
- 来源模拟视频。
- Clip 路径。
- YOLO candidate 摘要。
- 当前状态说明。
- VLM 初步结果，如果已有。
- VLM 判断理由或失败原因，如果有。
- 隐私状态。
- 完整性状态。
- 留存状态。

空状态：

```text
当前暂无待复核告警。
```

边界：

- 第一版不提供“通过”“拒绝”“提交审核结果”等按钮。
- 该模块只读，为后续人工审核流程预留位置。

### 6.6 案例回放与评估

目标：在一个模块里查看告警/事件回放案例和当前展示范围内的评估指标。

可点击案例范围：

- `confirmed_fall`
- `candidates`
- `need_human_review`

默认不展示：

- `rejected` 事件不出现在可点击列表中。
- `rejected` 视频默认不可打开。
- `rejected` 只进入评估统计，用于说明系统具备过滤误报能力。

案例列表字段：

- 告警/事件类别。
- 摄像头 ID。
- 区域/房间。
- YOLO 分数。
- VLM 结果。
- VLM 置信度。
- 片段时长。
- 告警时间。

筛选项：

```text
全部可展示案例
确认摔倒
待 VLM/人工复核
需人工复核
```

指标范围：

- 统计当前用于展示的事件集合。
- 可点击列表不包含 `rejected`。
- 指标中包含 `rejected` 数量或 VLM 拒绝数量。
- 不强调 `data/events` 的历史累计总量。

指标内容：

- 可展示案例数量。
- 确认摔倒数量。
- 待复核告警数量。
- 疑似摔倒告警数量。
- VLM 拒绝数量 / rejected 数量。
- YOLO 平均分。
- VLM 平均置信度。
- VLM 已复核数量。
- VLM 确认 / 拒绝 / 人工复核数量。

如果有人工标注 CSV 并已用于指标报告，可展示：

- Precision。
- Recall。
- F1。
- 起始时间 1000 ms 内准确率。
- 起始时间 2000 ms 内准确率。
- 平均起始时间绝对误差。

如果没有标注数据，展示：

```text
当前未提供人工标注文件，因此 Precision、Recall、F1 和时间准确率尚未计算。
```

### 6.7 技术方案

目标：说明系统如何实现摄像头风险监测。

展示链路：

```text
摄像头视频流
  -> YOLO 实时候选检测
  -> 事件前后帧缓存
  -> 疑似摔倒告警生成
  -> Video VLM 复核
  -> 摔倒事件记录 / 待复核告警 / 误报过滤
```

必须说明：

```text
当前原型使用本地视频模拟摄像头输入，后续可接入 RTSP 或真实摄像头。
```

已实现链路：

- 本地视频模拟摄像头输入。
- YOLO pose/person 模型生成疑似摔倒候选。
- EventBuffer 保存事件触发点前后的帧。
- ClipBuilder 写入事件视频片段和 JSON 元数据。
- SQLite 队列支持异步 VLM worker 复核。
- SQLite 事件状态优先作为前端分类依据。
- 指标报告汇总输出和评估结果。

后续安全扩展方向：

- RTSP / 真实摄像头接入。
- 持续实时视频画面。
- 事件视频加密。
- 人体区域隐私保护。
- 视频哈希校验。
- 多节点备份。
- 删除/篡改检测。
- 授权查看。

## 7. 数据归类规则

前端必须按照下面规则归类：

```text
confirmed_fall
  -> 实时监测高风险摄像头
  -> 告警中心
  -> 摔倒事件记录
  -> 案例回放与评估可点击案例
  -> 优先使用 SQLite status，JSON category 作为补充

candidates
  -> 实时监测疑似摔倒摄像头
  -> 告警中心
  -> 待复核告警
  -> 案例回放与评估可点击案例

need_human_review
  -> 实时监测待复核摄像头
  -> 告警中心
  -> 待复核告警
  -> 案例回放与评估可点击案例

rejected
  -> 默认不展示事件列表
  -> 不作为可点击案例
  -> 在案例回放与评估中统计数量
  -> 可作为“已过滤误报”指标
```

## 8. 摄像头建模规则

第一版摄像头模型从事件数据推导。

摄像头卡片生成：

- 收集所有事件中的 `camera_id`。
- 每个唯一 `camera_id` 生成一张摄像头卡片。
- 没有 `camera_id` 的事件不生成摄像头卡片，但仍可进入对应事件列表。
- 没有任何事件时，实时监测页显示空状态。

摄像头状态优先级：

```text
confirmed_fall > need_human_review > candidates > normal
```

对应显示文案：

```text
confirmed_fall      -> 已确认摔倒
need_human_review   -> 待复核
candidates          -> 疑似摔倒
normal              -> 正常
```

摄像头卡片画面：

- 第一版不播放实时流。
- 使用占位画面。
- 可提供“查看最近告警回放”的入口。

## 9. 数据契约

前端由 `app.py` 提供本地 API。

推荐接口：

```text
GET /api/cameras
GET /api/alerts
GET /api/fall-events
GET /api/review-alerts
GET /api/showcase
GET /api/events/<event_id>
GET /api/evaluation
GET /api/config
GET /media/<path>
```

### 9.1 `/api/cameras`

返回实时监测模块的摄像头卡片、顶部状态栏和最新告警摘要。

响应结构：

```json
{
  "summary": {
    "camera_count": 0,
    "risk_camera_count": 0,
    "confirmed_fall": 0,
    "review_alerts": 0,
    "candidate_alerts": 0,
    "rejected": 0
  },
  "cameras": [],
  "latest_alerts": [],
  "queue": {},
  "simulation_note": "当前原型使用本地视频模拟摄像头输入，因此本页暂不展示持续实时画面；接入 RTSP 或真实摄像头后，可替换为实时视频流。"
}
```

### 9.2 `/api/alerts`

返回告警中心列表。

响应结构：

```json
{
  "alerts": []
}
```

其中 `alerts` 不包含 `rejected`。

### 9.3 `/api/fall-events`

返回“摔倒事件记录”列表。

响应结构：

```json
{
  "events": [],
  "empty_message": "当前暂无已确认摔倒事件。"
}
```

### 9.4 `/api/review-alerts`

返回“待复核告警”列表。

响应结构：

```json
{
  "alerts": [],
  "empty_message": "当前暂无待复核告警。"
}
```

### 9.8 `/api/config`

### 9.9 `/media/<path>`

提供已保存告警/事件视频回放片段。必须防止路径穿越，只允许提供项目目录或配置事件输出目录下的文件。

## 10. 视觉方向

界面气质应当是：

- 专业。
- 技术可信。
- 克制。
- 面向项目展示和摄像头监测。
- 适合科研团队对合作方展示。

推荐布局：

- 首页是项目介绍页，突出项目价值、应用场景、已实现能力和后续扩展方向。
- 实时监测页是摄像头监控工作台。
- 实时监测页顶部状态栏展示当前系统状态。
- 摄像头监测区是实时监测页主体。
- 摄像头卡片使用占位画面，不假装实时流。
- 最新告警侧栏突出风险事件。
- 告警中心、摔倒事件记录、待复核告警都使用列表 + 回放详情布局。
- 案例回放与评估使用“案例列表 + 指标卡片”的组合布局。
- 技术方案放在后面，支撑讲解。

避免：

- 纯营销落地页。
- 把首页做成空泛宣传页。
- 把实时监测页做成视频文件检测结果列表。
- 为未实现的安全模块放置假按钮。
- 文案宣称隐私保护或防篡改已经完成。
- 把 rejected 事件默认放进可点击案例列表。
- 假装当前本地视频模拟输入是持续实时摄像头流。

## 11. 成功标准

第一版前端完成时应满足：

- `python app.py` 能启动本地服务。
- 浏览器无需前端构建步骤即可打开页面。
- 首页是项目介绍页。
- “实时监测”作为同级导航模块可点击查看。
- 七个部分都可以访问。
- 实时监测页的摄像头卡片从事件数据中的 `camera_id` 自动生成。
- 摄像头卡片使用占位画面，并说明当前为模拟摄像头。
- 摄像头卡片状态按最高风险优先级显示。
- `confirmed_fall` 能显示在“摔倒事件记录”中。
- `candidates` 和 `need_human_review` 能显示在“待复核告警”中。
- 告警中心显示 confirmed/candidates/review 类型告警。
- 两个事件类模块都能点击查看视频回放详情。
- `rejected` 不出现在可点击案例列表中。
- `rejected` 数量能出现在“案例回放与评估”的指标里。
- `data/records.db` 存在时，事件状态优先使用 SQLite。
- 数据库不存在时，JSON `category` 仍可作为回退。
- 页面不会启动模型推理，也不会修改事件数据。
- UI 明确区分已实现能力和后续扩展方向。
